# 虚拟下位机规范文档使用指南

> 目的：本指南不是新的业务规则，而是项目所有规范、计划和参考资料的“目录地图”。
> 不确定先看哪份文件、收到新的 PLC/中台资料后该改哪里时，先看这里。

## 1. 先看这张地图

```text
需求与现场资料（只读参考）
reference/requirements/
        │
        ▼
项目合同（当前实现必须遵守）
docs/ + openapi/ + config/
        │
        ▼
Spec Kit 规格与实施计划（说明为什么做、怎样分步做）
specs/001-virtual-line-simulation/
        │
        ▼
未来 C# 代码与自动测试
src/ + tests/
```

简单理解：

- `reference/requirements` 是客户、协议和节拍表的原始资料，**不直接修改**。
- `docs`、`openapi`、`config` 是目前已经整理出的“项目合同”。写代码必须遵守它们。
- `specs` 是把合同拆成需求、设计和开发任务的工作说明书。
- `src`、`tests` 将在第二阶段才出现，分别放程序和自动测试。

## 3. 每份规范文档分别负责什么

### 3.1 项目入口与交接

| 文件 | 负责回答什么 | 什么时候看/改 |
|---|---|---|
| `README.md` | 项目在哪里、目录如何分工、当前到哪个阶段 | 新加入项目、目录变化、阶段完成时 |
| `PROJECT_HANDOFF.md` | 已完成什么、未完成什么、服务器环境与下一步 | 每次新对话或阶段结束时 |
| `docs/虚拟下位机CSharp保姆级开发指导书.md` | 从零开始的整体开发路线 | 想理解“为什么按这个顺序做”时；通常不作为接口真相来源 |

### 3.2 中台与下位机的接口合同

| 文件 | 负责回答什么 | 谁需要它 |
|---|---|---|
| `docs/plc-field-table.md` | 每个业务字段的方向、类型、默认值、谁写、何时清除、超时怎么处理 | 中台、下位机、PLC 工程师 |
| `openapi/openapi.yaml` | HTTP 地址、请求 JSON、响应 JSON、错误码 | 中台开发和 C# API 开发 |
| `specs/001-virtual-line-simulation/contracts/http-signalr-contract.md` | HTTP 查询/命令与 SignalR 实时事件各自负责什么 | 中台实时页面和 C# API 开发 |

它们之间的关系：

```text
字段表：字段本身是什么意思
      │
      ▼
OpenAPI：这些字段如何通过 HTTP 传递
      │
      ▼
SignalR 合同：状态、报警、工件变化怎样实时通知
```

### 3.3 设备运行规则

| 文件 | 负责回答什么 | 核心内容 |
|---|---|---|
| `docs/state-machine.md` | 哪些状态能发什么命令，命令后去哪里 | Idle、Running、Paused、Stopped、Alarm；Stop 后 Aborted；Home 后才可重新启动 |
| `docs/alarm-catalog.md` | 出现异常时报警叫什么、严重程度、如何恢复 | 命令、PLC、安全、资源、OP10～OP80、回原点、配置报警 |
| `config/takt-plan.json` | 工序顺序、资源、时长、超时与投料规则 | OP10～OP80、Robot 等 9 类资源、临时仿真时间 |

它们之间的关系：

```text
状态机：设备此刻能不能做某件事
      │
      ├── 报警目录：出错后进入何种异常处理
      │
      └── 节拍配置：运行时工件按怎样的工序和资源推进
```

## 4. 什么是“当前已冻结”，什么是“待确认”

### 当前实现必须遵守的基线

1. 中台不能直接访问 PLC 地址，只能调用版本化 HTTP API 与订阅 SignalR 事件。
2. 设备状态是 `Idle`、`Running`、`Paused`、`Stopped`、`Alarm`。
3. 命令有唯一 `commandId`；相同命令重发不能重复执行。
4. Stop 后所有在制品是 `Aborted`；完成 Home 后才回到 Idle。
5. 允许多件工件并行，但容量为一的资源不能同时分给两件工件。
6. 软件不替代急停、安全门、光栅等硬件安全回路。

### 不能当成最终事实的内容

- PLC 品牌、网络地址、寄存器/数据块地址、字节序和读写周期；
- 真实设备的暂停/停止时序；
- SN 格式、重复 SN 与读码失败策略；
- 52 项测量字段、螺纹结果字典、最终 OK/NG 汇总规则；
- 最终 PLC 报警码、权限认证、事件重放和历史数据保存策略；
- 每道工序的精确现场时长。

这些内容出现时，必须根据本文第 5 节受控修改，不可直接把临时值当作现场事实。

## 5. 收到真实中台、PLC 或设备资料后，应该改哪里？

```text
收到资料
  │
  ├─ 字段名、类型、方向、PLC 地址、读写时序
  │    → docs/plc-field-table.md
  │
  ├─ HTTP 请求/响应字段或地址
  │    → openapi/openapi.yaml
  │
  ├─ SignalR 事件名、事件 JSON、重连规则
  │    → specs/.../contracts/http-signalr-contract.md
  │
  ├─ 启动、暂停、停止、复位、回原点的真实语义
  │    → docs/state-machine.md
  │
  ├─ 现场故障码、恢复流程、是否必须回原点
  │    → docs/alarm-catalog.md
  │
  ├─ 工位、资源、准确节拍、超时
  │    → config/takt-plan.json
  │
  └─ 52 项结果、螺纹结果、最终 OK/NG 规则
       → docs/plc-field-table.md
       → openapi/openapi.yaml
       → specs/.../data-model.md
       → specs/.../spec.md 和 tasks.md（若实现范围改变）
```

### 推荐修改步骤

1. 保留供应商/中台给出的原始资料在 `reference/requirements/` 或新建的受控参考目录。
2. 对照本文第 5 节，找出受影响的规范文件。
3. 若是正式的真实设备联调，建立新的 Spec Kit 规格目录，例如
   `specs/002-middle-platform-plc-icd/`，记录来源、变更原因、兼容性和任务。
4. 先更新合同与测试，再更新 C# 代码。
5. 运行测试后 `git add`、`git commit`、`git push`，保证 Gitee 有完整变更历史。

## 6. 当前必须先处理的 P0 问题

在进入第二阶段前，节拍配置存在一处需要修订的矛盾：

```text
当前 OP10～OP70 临时时长合计：100 秒
当前投料规则文字：第一件约 65 秒进入 OP80 后放行第二件
```

两者不能同时成立。原始节拍资料说明“首件完整约 125 秒、稳定约 65 秒/件”，但各 OP 的精确时长尚未冻结。

处理原则：保留 125 秒与 65 秒为已有依据；把 OP 分时和“进入 OP80”的触发条件改成一致的临时模型，或明确标记为待确认。完成这项小修订后，再开始第二阶段。

## 7. 第二阶段开始后怎样使用本指南

开发时可以按下面顺序查：

```text
想写一个命令接口？
  → state-machine.md
  → plc-field-table.md
  → openapi/openapi.yaml
  → tasks.md 中对应任务

想写工件流转？
  → takt-plan.json
  → state-machine.md
  → data-model.md
  → tasks.md 中对应任务

想写报警？
  → alarm-catalog.md
  → state-machine.md
  → data-model.md
  → tasks.md 中对应任务
```
