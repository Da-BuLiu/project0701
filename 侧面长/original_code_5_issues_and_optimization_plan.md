# 原始代码 5 个问题分析与算法优化方向

本文档只分析当前根目录中的原始脚本：

```text
measure_lengths.py
```

不再引用已经删除的 `output/optimized_results` 版本。

## 1. 总体结论

当前原始算法的问题不是某一个 OpenCV 参数没有调好，而是测量对象选错了。

代码现在主要测的是图像中的“高亮反光长条”的左右端点，而不是零件真实的物理端面或几何轮廓。只要不同位置的反光形态发生变化，例如 Pos3 的高亮区域比真实零件短，算法就会稳定地测到错误的长度。

所以，后续优化不能只继续调整阈值、投影百分位数、形态学核大小或置信度公式。真正的方向应该是：

1. 高亮反光条只用于粗定位 ROI 和估计零件姿态。
2. 最终长度端点必须来自零件真实端面或外轮廓。
3. 测量应沿人工标注线或零件轴线附近进行。
4. 标定应支持不同位置或多张样本，而不是只用一张 Pos1 图。
5. 异常判定应使用高精测量所需的绝对误差、公差和几何质量指标。

## 2. 当前原始算法流程

当前原始代码的大致流程如下：

1. 读取图像为灰度图。
2. 用 `detect_bright_side_band()` 检测图像中的亮色长条。
3. 根据亮色长条的连通域外接矩形得到粗略左右边界。
4. 在整个亮条高度范围内做一维亮度投影。
5. 在一维亮度曲线上寻找左右亮度跳变点。
6. 用 `s1_label` 的人工标注线建立偏移量。
7. 把该偏移量应用到所有样品、所有位置。
8. 输出长度、角度、置信度和异常信息。

关键函数包括：

| 函数 | 作用 | 当前风险 |
|---|---|---|
| `detect_bright_side_band()` | 找高亮长条 | 找的是反光，不是零件真实边界 |
| `sample_percentile_projection_profile()` | 对亮条区域做一维亮度投影 | 会丢失二维空间信息 |
| `detect_subpixel_side_edges()` | 从一维曲线中找左右端点 | 实际没有使用测量线参数 |
| `build_calibration()` | 用人工标注线计算偏移 | 只基于单张 Pos1 标定 |
| `detect_length()` | 对每张图输出长度和状态 | 异常门限过宽 |

## 3. OpenCV 和算法术语解释

### 3.1 灰度图

灰度图是只有亮度信息的图像。每个像素通常是 `0..255`：

- `0` 表示黑色。
- `255` 表示白色。
- 中间值表示不同程度的灰。

当前代码用灰度图做检测，因此颜色信息不会参与判断。

### 3.2 阈值分割

阈值分割是把灰度图变成黑白图的方法。

例如设阈值为 `120`：

- 灰度值大于 `120` 的像素变成白色。
- 灰度值小于等于 `120` 的像素变成黑色。

这样可以把“亮的区域”和“暗的区域”分开。

### 3.3 Otsu 阈值

`Otsu` 是一种自动找阈值的方法。

它会分析整张图的灰度分布，自动找一个阈值，让分成的两类像素差异尽量大。简单说，它试图自动回答：

```text
哪些像素应该算亮？
哪些像素应该算暗？
```

在当前代码中：

```python
threshold, binary = cv2.threshold(gray, 0, 255, cv2.THRESH_BINARY + cv2.THRESH_OTSU)
```

这表示代码没有手动给阈值，而是交给 Otsu 自动决定。

风险是：Otsu 找到的是亮暗分界，不知道什么是零件，也不知道什么是反光。如果反光特别亮，它会自然偏向检测反光。

### 3.4 二值图和 mask

二值图就是只有黑白两种值的图。

当前代码里：

```python
mask = (binary > 0).astype(np.uint8)
```

这一步把白色像素记为 `1`，黑色像素记为 `0`。这个 `mask` 可以理解成一张“亮区域地图”。

### 3.5 形态学闭运算

形态学是 OpenCV 中处理二值图形状的一类方法。

闭运算 `MORPH_CLOSE` 通常可以理解为：

1. 先膨胀：让白色区域变粗、变大。
2. 再腐蚀：把白色区域缩回去。

它常用于：

- 填补亮区域中的小黑洞。
- 连上亮区域中的小断裂。
- 让碎裂的亮块看起来更完整。

当前代码中：

```python
close_kernel = cv2.getStructuringElement(cv2.MORPH_RECT, (close_w, 7))
mask = cv2.morphologyEx(mask, cv2.MORPH_CLOSE, close_kernel)
```

`kernel` 是形态学操作的窗口，可以理解成刷子大小。这里是一个横向较长、纵向较短的矩形刷子，适合把横向亮条连接起来。

风险是：这一步会让反光条变得更连续，但不会让它变成真实零件边界。

### 3.6 连通域

连通域是二值图中连在一起的一片白色像素区域。

如果一张图里有多个白色区域，比如高光、划痕、背景亮点，连通域检测会把它们分别编号。

当前代码使用：

```python
num_labels, labels, stats, _centroids = cv2.connectedComponentsWithStats(mask, 8)
```

`stats` 里包含每个连通域的：

- `x`：外接矩形左上角 x 坐标。
- `y`：外接矩形左上角 y 坐标。
- `w`：外接矩形宽度。
- `h`：外接矩形高度。
- `area`：白色像素面积。

### 3.7 bbox

`bbox` 是 bounding box 的缩写，意思是外接矩形。

例如某个亮区域最左是 `x=100`，最右是 `x=500`，最上是 `y=200`，最下是 `y=260`，它的 bbox 就大致是：

```text
x = 100
y = 200
w = 400
h = 60
```

当前代码用亮色连通域的 bbox 来表示亮条的大致位置。

风险是：bbox 描述的是亮区域的范围，不是零件物理轮廓的范围。

### 3.8 长宽比 aspect ratio

长宽比就是：

```text
宽度 / 高度
```

当前代码要求候选区域的长宽比足够大：

```python
aspect = w / max(h, 1)
if width_ratio < 0.35 or aspect < 4.0 or area_ratio < 0.015:
    continue
```

这表示代码倾向于选择“横向很长”的亮区域。

这对找反光长条有效，但对找真实端面没有直接帮助。

### 3.9 ROI

`ROI` 是 region of interest，感兴趣区域。

在图像算法中，通常不会直接在整张图上做复杂检测，而是先找到一个大致区域，再在这个区域里精细处理。

当前代码实际上把亮条 bbox 当作 ROI 使用。

合理的优化方向是：亮条可以继续作为 ROI 参考，但最终测量不能只依赖亮条端点。

### 3.10 投影 profile

`profile` 可以理解为一条一维曲线。

当前代码把二维图像区域压缩成一条横向亮度曲线：

```python
profile = np.percentile(roi, percentile, axis=0)
```

`axis=0` 表示对每一列像素做统计。最终每个 x 坐标只有一个亮度值。

例如某个 x 坐标下有很多个 y 像素，代码会取这些像素的第 98 分位值，作为这一列的亮度。

### 3.11 百分位数 percentile

第 98 分位数可以理解为：把一组数从小到大排序后，位于 98% 位置的值。

它接近最大值，但比最大值稍微稳一点。

在当前算法中，使用 98 分位意味着：

```text
只要这一列的某些高度位置非常亮，这一列就容易被认为是亮的。
```

这对检测反光很敏感，但对真实几何边界不稳定。

### 3.12 亚像素 subpixel

普通像素坐标是整数，例如 `x=100`、`x=101`。

亚像素是小数坐标，例如 `x=100.37`。它试图通过插值和曲线分析，在两个像素之间估计更精细的位置。

亚像素本身不是问题。问题是：如果用于亚像素定位的一维 profile 本身来自反光条，那么得到的只是更精细的反光边缘，而不是更精细的真实端面。

### 3.13 梯度 gradient

梯度表示亮度变化速度。

如果图像从暗突然变亮，梯度很大。边缘检测通常就依赖这种突变。

当前代码中：

```python
gradient = np.gradient(profile)
```

它是在一维亮度曲线上计算梯度，用来辅助判断边缘强度。

### 3.14 置信度 confidence

置信度是代码自己计算出来的质量分数。

当前代码的置信度主要由亮带宽度、长宽比、面积、填充率、边缘强度、对比度等指标组成。

风险是：这些指标可以说明“反光条检测得是否清楚”，但不能证明“零件真实长度是否测准”。

## 4. 问题 1：`detect_bright_side_band()` 找的是高亮反光区域，不是零件真实几何边界

代码位置：

```text
measure_lengths.py:154
```

相关逻辑：

```python
threshold, binary = cv2.threshold(gray, 0, 255, cv2.THRESH_BINARY + cv2.THRESH_OTSU)
mask = (binary > 0).astype(np.uint8)
mask = cv2.morphologyEx(mask, cv2.MORPH_CLOSE, close_kernel)
num_labels, labels, stats, _centroids = cv2.connectedComponentsWithStats(mask, 8)
```

这段代码的本质是：

1. 自动找出亮像素。
2. 把亮像素连接成更完整的区域。
3. 找出最长、最像横向亮条的白色连通域。

它没有做的事情包括：

- 没有检测零件外轮廓。
- 没有检测左右端面。
- 没有判断反光条是否覆盖了完整零件长度。
- 没有区分真实材料边界和局部高光。

因此，只要图像中的高亮区域与真实零件边界不一致，检测结果就会偏离真实长度。

例如 Pos3 中如果反光条左右端比真实端面短，代码就会把短的反光条当作零件长度来源，最终稳定输出偏短结果。

这个问题属于测量对象错误，不是阈值参数错误。

## 5. 问题 2：`sample_percentile_projection_profile()` 把二维图像压缩成一维亮度曲线

代码位置：

```text
measure_lengths.py:244
```

相关逻辑：

```python
y0 = max(0, int(base.bbox_y))
y1 = min(gray.shape[0], int(base.bbox_y + base.bbox_h))
roi = gray[y0:y1, :].astype(np.float32)
profile = np.percentile(roi, percentile, axis=0).astype(np.float32)
```

这里的 `roi` 是从亮条 bbox 的顶部到底部取出的图像区域。然后对每一列取第 98 分位亮度。

这个方法的问题是，它把二维图像中的上下空间信息全部压扁了。

原本图像里可能有这些情况：

- 上半部分亮，下半部分暗。
- 某一高度有划痕。
- 某一高度有局部反光。
- 端部一部分亮、一部分暗。
- 真实端面是斜的或带倒角。

经过 98 分位投影之后，每个 x 坐标只剩一个数。算法不再知道这个亮度来自哪个 y 高度，也不知道它是否位于人工标注线附近。

这会导致两个风险：

1. 远离测量线的局部亮点也会影响端点。
2. 真实端面附近的暗区可能被其他高度的亮点掩盖。

因此，这种 profile 更适合检测“某一列是否存在高亮反光”，不适合直接作为真实长度测量依据。

## 6. 问题 3：`detect_subpixel_side_edges()` 接收测量线参数，但开头直接丢弃

代码位置：

```text
measure_lengths.py:332
```

相关逻辑：

```python
def detect_subpixel_side_edges(
    gray: np.ndarray,
    base: BaseDetection,
    measurement_slope: float,
    y_mid: float,
) -> EdgeDetection:
    del measurement_slope, y_mid
    profile = sample_percentile_projection_profile(gray, base, percentile=98.0)
```

函数参数中的 `measurement_slope` 和 `y_mid` 理论上非常重要。

`measurement_slope` 是测量线斜率。它表示人工标注线或零件长轴的倾斜程度。

`y_mid` 是测量线经过图像中间位置时的高度。

如果真正沿标注线测量，算法应该使用这两个参数，在标注线附近构造一条窄带，然后只分析窄带中的像素。

但当前代码直接：

```python
del measurement_slope, y_mid
```

这意味着这两个参数完全没有参与亚像素边缘检测。

后续代码仍然使用整块亮条投影：

```python
profile = sample_percentile_projection_profile(gray, base, percentile=98.0)
```

所以当前所谓的亚像素边缘检测并不是沿人工标注线或零件轴线进行的。它只是对整块高亮带做了一维投影，再在曲线上找左右跳变点。

这对高精测量非常危险，因为测量线位置不同，端点可能不同。尤其当端面倾斜、反光不均、局部暗区存在时，整块投影得到的边缘不一定对应人工标注线上的真实端点。

## 7. 问题 4：`build_calibration()` 只用单张 `s1_label` 标定图，不能代表所有位置

代码位置：

```text
measure_lengths.py:378
```

相关逻辑：

```python
p1, p2 = read_label_line(label_json)
manual_length = math.hypot(dx, dy)
manual_slope = dy / dx
gray = load_gray(label_image)
base = detect_bright_side_band(gray)
edge = detect_subpixel_side_edges(gray, base, manual_slope, y_mid)
```

标定阶段会计算：

- 人工标注长度。
- 人工标注线斜率。
- 人工端点相对亮条 bbox 左右边缘的偏移。
- 人工端点相对亚像素亮边缘的偏移。
- 人工线在亮条高度中的相对位置。

这些偏移之后会用于所有图片。

问题是：单张 Pos1 图中的“人工标注端点”和“自动检测亮带端点”的关系，不一定适用于 Pos2 和 Pos3。

不同位置可能存在：

- 反光角度不同。
- 光带长短不同。
- 端部暗区不同。
- 表面划痕位置不同。
- 零件姿态略有变化。
- 镜头或夹具造成的局部成像差异。

如果 Pos3 的反光条本身比 Pos1 短，那么 Pos1 中学到的偏移量无法修正 Pos3 的物理端面。它只会把 Pos1 的经验强行套到 Pos3 上。

这类方法可以修正固定偏移，但不能修正跨位置的反光形态变化。

## 8. 问题 5：异常门限过宽，20 px 级误差仍会被判为 OK

代码位置：

```text
measure_lengths.py:608
```

相关逻辑：

```python
if not (0.75 <= ratio <= 1.25):
    anomalies.append(...)
```

这里的 `ratio` 是：

```text
当前测量长度 / 标定参考长度
```

代码只要求长度在参考长度的 75% 到 125% 之间。

如果参考长度约为 2737 px，那么允许范围大约是：

```text
下限：2737 * 0.75 = 2052.75 px
上限：2737 * 1.25 = 3421.25 px
```

也就是说，只要测出来不是离谱到少了几百像素，都会被认为没有严重错误。

但对高精度测量来说，20 px 已经很大。如果换算后约为 0.144 mm，那么这可能已经超过工艺允许误差。

因此，`0.75..1.25` 这种比例门限只能发现灾难级错误，不能用于精密测量的 OK 判定。

更合理的门限应该包括：

- 绝对像素误差，例如超过 3 px 警告，超过 8 px 错误。
- 毫米误差，例如超过工艺公差即错误。
- 重复性误差，例如同组 20 张图的极差或标准差超限。
- 几何质量错误，例如边缘点太少、拟合残差太大。

## 9. 原始算法为什么会在 Pos3 上稳定偏短

根据当前流程，如果 Pos3 的高亮反光条本身短于真实零件，那么算法会发生以下连锁反应：

1. Otsu 阈值检测到的是短的高亮区域。
2. 闭运算会把这个短高亮区域变得更连续。
3. 连通域检测会把它选为主亮条。
4. bbox 左右边界来自这个短亮条。
5. 98 分位投影进一步强调高亮区域。
6. 亚像素边缘检测在反光曲线上找左右边缘。
7. 单张 Pos1 标定偏移无法补偿 Pos3 的反光缩短。
8. `0.75..1.25` 门限太宽，导致偏短结果仍然是 OK。

所以 Pos3 的问题不是随机噪声，而是系统性误差。只要反光形态稳定偏短，算法就会稳定测短。

## 10. 优化方向 1：沿测量轴线构造局部窄带 profile

当前代码有 `measurement_slope` 和 `y_mid`，但没有真正用它们做边缘检测。

优化时应该沿测量线构造局部窄带。

可以理解为：

1. 先确定一条中心线，这条线代表人工标注线或零件长轴。
2. 在线的上下方向各取一定宽度，例如 10 到 30 px。
3. 只在这条窄带内采样图像亮度。
4. 根据这条窄带生成 profile 或边缘点。

这样可以避免远离测量线的划痕、反光、暗区影响长度。

这一步能解决的问题是：让测量更接近人工标注位置。

它不能单独解决的问题是：如果测量线附近看到的仍然只是反光端点，而不是物理端面，误差仍可能存在。

## 11. 优化方向 2：检测零件真实端面，而不是检测反光亮带端点

这是最关键的优化方向。

真实长度应该来自零件左右物理端面。端面是零件实际结束的位置，不应该由反光条决定。

可考虑的方法包括：

### 11.1 边缘检测

边缘检测寻找图像中亮度快速变化的位置。

常用方法：

- `Sobel`：计算图像在 x 或 y 方向的亮度梯度。
- `Canny`：更完整的边缘检测算法，包含降噪、梯度计算、非极大值抑制和双阈值连接。
- 一维梯度：沿测量轴线或垂直于端面的方向采样，寻找最大亮度变化点。

如果零件端面和背景之间存在稳定明暗变化，边缘检测可以找到物理边界。

### 11.2 端面线拟合

不要只找一个边缘点。应该在左端和右端分别找多个边缘点，然后拟合端面直线。

可选算法：

- 最小二乘拟合：适合边缘点干净、离群点少的情况。
- RANSAC：适合有划痕、噪声、误检点时使用。
- Hough Line：适合从边缘图中寻找明显直线。

拟合出左右端面线后，再用零件轴线与端面线求交点。两个交点之间的距离就是更接近真实物理长度的测量值。

### 11.3 轴线与端面求交

当前算法主要找的是左右 x 坐标，然后用斜率补出 y 坐标。

更稳的几何模型应该是：

1. 估计零件轴线。
2. 检测左端面线。
3. 检测右端面线。
4. 求轴线与左端面线的交点。
5. 求轴线与右端面线的交点。
6. 计算两个交点距离。

这样测量点有明确几何意义，而不是来自反光曲线的左右跳变。

## 12. 优化方向 3：图像姿态归一化

如果零件在图中有轻微倾斜，直接在原图上分析会复杂一些。

可以先估计零件长轴方向，然后把 ROI 旋转到水平。这叫姿态归一化，也可以用仿射变换实现。

归一化后的好处：

- 零件长度方向接近水平。
- 左右端面接近竖直。
- 端面搜索窗口更容易定义。
- profile 和边缘检测更稳定。

需要注意：旋转会引入插值，因此最好保留原图坐标和变换矩阵，最终结果可以再映射回原始图像坐标。

## 13. 优化方向 4：使用亮带作为辅助，而不是最终测量依据

亮带并非完全无用。它仍然可以用来：

- 粗定位零件区域。
- 排除大部分背景。
- 估计零件大致方向。
- 给端面检测提供搜索范围。

但最终端点不应该直接等于亮带端点。

更合理的结构是：

```text
亮带检测 -> ROI 和姿态估计 -> 端部区域搜索 -> 端面边缘检测 -> 端面线拟合 -> 轴线求交 -> 长度输出
```

这比当前结构更符合物理测量逻辑。

## 14. 优化方向 5：多位置标定或多样本标定

当前只用一张 `s1_label` 做标定。

更稳的标定方式包括：

### 14.1 每个 position 单独标定

如果 Pos1、Pos2、Pos3 的成像状态不同，应分别提供标注线：

```text
pos1_label.json
pos2_label.json
pos3_label.json
```

每个位置用自己的参考线、偏移和容差。

### 14.2 多张样本统计标定

如果每个位置有多张可靠图像，可以计算平均值和波动范围。

这样能得到：

- 平均长度。
- 标准差。
- 正常波动范围。
- 异常判定阈值。

### 14.3 像素到毫米标定

如果最终需要毫米单位，应建立：

```text
px_per_mm = 像素长度 / 实际毫米长度
```

然后异常判断应基于真实工艺公差，例如：

```text
长度误差 > 0.02 mm -> warning
长度误差 > 0.05 mm -> error
```

具体数值应由实际工艺要求决定。

## 15. 优化方向 6：重做异常判定和质量评估

当前 `0.75..1.25` 太宽，不适合精密测量。

建议至少增加以下检查：

### 15.1 长度绝对偏差

例如：

```text
abs(length_px - reference_length_px) > 3 px -> warning
abs(length_px - reference_length_px) > 8 px -> error
```

如果有毫米标定，则应换成毫米公差。

### 15.2 端面拟合质量

如果使用端面线拟合，应输出：

- 左端面边缘点数量。
- 右端面边缘点数量。
- 拟合残差。
- RANSAC 内点比例。
- 左右端面角度是否合理。

拟合残差表示边缘点离拟合直线的平均距离。残差越大，说明端面越不稳定或检测越差。

### 15.3 重复性检查

同一样品、同一位置有多张重复图时，应统计：

- 最小值。
- 最大值。
- 极差。
- 平均值。
- 标准差。
- 变异系数。

如果同组内部波动超过门限，应标记为异常。

### 15.4 fallback 状态降级

如果精细端面检测失败，被迫退回 bbox 或亮带端点，结果不应继续作为高可信 OK。

建议规则：

```text
使用 fallback -> 至少 warning
fallback 且偏差较大 -> error
```

## 16. 建议的下一版算法结构

建议下一版按以下结构实现：

```text
1. 读取灰度图
2. 检测亮带，只用于定位 ROI
3. 根据亮带或标注线估计零件长轴
4. 在 ROI 内做光照归一化和降噪
5. 在左右端部搜索窗口中检测边缘点
6. 分别拟合左端面线和右端面线
7. 用长轴线与左右端面线求交点
8. 计算两个交点之间的长度
9. 输出长度、端面质量、拟合残差、异常状态
10. 汇总重复性报告
```

该结构中，高亮反光条不再是测量对象，而只是定位工具。

## 17. 实施优先级

建议按以下顺序推进：

1. 先修复 `detect_subpixel_side_edges()` 不使用 `measurement_slope` 和 `y_mid` 的问题，让测量至少发生在标注线附近。
2. 增加严格异常门限，避免 20 px 级错误继续被标为 OK。
3. 增加输出诊断字段，例如 profile 类型、边缘宽度、长度绝对偏差、是否 fallback。
4. 开发端面检测版本，把最终端点从反光条端点切换到物理端面。
5. 增加每个 position 的标定或多样本标定。
6. 用现有 120 张图验证每组均值、标准差、极差和异常分布。

## 18. 最终判断

当前原始代码的 5 个问题可以归纳为一句话：

```text
算法把“反光亮带”当成了“零件真实长度”，并且异常门限没有阻止错误结果进入 OK 报告。
```

因此，真正的优化目标不应是让反光条检测更精细，而应是让最终测量点具备明确的物理意义。

推荐目标是：

```text
亮带定位 + 轴线估计 + 端面检测 + 几何求交 + 多位置标定 + 严格质量判定
```

只有这样，才能从算法层面解决 Pos3 这类由反光形态变化导致的系统性误差。
