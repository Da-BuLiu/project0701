# Research: 137 壳体虚拟下位机基线

## Decision: 分层为 API、Domain 和 Infrastructure 三个项目

**Rationale**: 中台接口、状态机/调度规则和 PLC/设备通信变化频率不同。领域层保持无现场协议依赖，
可以独立测试；基础设施层同时容纳当前仿真和未来真实 PLC 适配器。

**Alternatives considered**:

- 单一 Web 项目：起步快，但真实 PLC 到位时会将通信细节和中台接口耦合。
- 为每个工位建立单独服务：当前范围过早增加部署与诊断复杂度。

## Decision: 以状态机和命令记录作为设备控制的唯一入口

**Rationale**: `commandId` 幂等和明确的状态转换可抵御网络重试与并发控制请求，并使中台的“已受理”
和“动作完成”语义可区分。

**Alternatives considered**:

- 让接口直接改状态字段：无法防止非法转换和重复执行。
- 让中台读写 PLC 位地址：破坏稳定接口，也无法离线仿真。

## Decision: 用单个后台调度器推进工件，资源以原子方式获取

**Rationale**: 单一调度决策点简化暂停、停止和资源互斥；每个工序开始前一次性检查其全部资源，避免
“只拿到一半资源”导致死锁或错误占用。

**Alternatives considered**:

- 每件工件各自启动独立循环：并发控制较难验证，容易竞争同一资源。
- 强制全线单件串行：与 65 秒稳定生产节拍不符。

## Decision: 节拍配置必须经启动时验证，并保留临时参数标识

**Rationale**: 已知只有首件 125 秒和稳定 65 秒投料间隔；各工序分时仍是仿真估计。配置校验能在
错误数据进入调度前阻止启动，也让现场数据到位后无需改业务代码。

**Alternatives considered**:

- 将时长写入代码常量：无法审计或安全调整。
- 伪造精确现场时长：会误导中台联调和验收预期。

## Decision: 基线使用进程内仓储，预留持久化接口

**Rationale**: 本阶段重点是业务行为与中台合同，进程内状态可降低首个可运行版本的复杂度；工件、命令、
报警和事件通过接口访问，后续可接 SQLite 而不改变领域服务或 API。

**Alternatives considered**:

- 立即接入 SQLite：会在状态机与多工件调度尚未验证前引入模式迁移、备份和恢复范围。
- 不定义仓储边界：未来持久化会侵入状态机与调度器。

## Decision: 用 HTTP 查询/命令和 SignalR 事件分别覆盖拉取与推送

**Rationale**: 中台可在启动或重连后拉取权威快照，并在生产中订阅增量事件。事件仅为通知，不能替代
查询快照；这有助于处理临时断线与消息遗漏。

**Alternatives considered**:

- 只用轮询：状态及时性和调试体验较差。
- 只用推送：重连后难以恢复完整状态。

## Deferred External Decisions

真实 PLC 协议与地址、暂停/停止的现场安全时序、SN 与螺纹结果字典、52 项判定、身份鉴权、事件重放
保留期、SQLite 与原始文件留存策略均需要相应供应方或需求方确认。这些不阻断模拟基线，但阻断真实设备
适配器和最终验收。
