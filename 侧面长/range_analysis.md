# 静态重复性、像素到毫米换算与当前 JSON 结果解释

本文只解释概念和当前报表口径，不优化代码。

当前目录里的相关输出主要是：

- `output/measurements.csv`：每张图片的单次测量明细。
- `output/static_repeatability.csv`：按 `sample_id + position` 分组后的静态重复性统计。
- `output/repeatability_report.json`：JSON 总报告，里面的 `repeatability` 与 `static_repeatability.csv` 对应。

需要先把三个概念分开：

| 概念 | 比较对象 | 回答的问题 |
|---|---|---|
| 检出率 | 是否能输出结果 | 每件产品、每个尺寸项有没有测出来 |
| 静态重复性 | 同一对象、同一位置、同一条件下重复拍/重复测 | 算法稳定不稳定 |
| 准确度/系统误差 | 算法测量值 vs 标准值/真值/计量仪器值 | 算法测得准不准 |
| 位置间一致性/再现性 | 同一对象在不同位置、不同工位、不同姿态下的测量均值 | 换位置后是否还一致 |

你现在疑惑的核心就在这里：`length_range_px` 很小，只能说明同一组连拍的重复性好；不同 `pos` 的 `length_mean_px` 相差接近 20px，是另一类问题，不能被组内重复性抵消。

## 1. 验收标准里的三个重复性等级是什么

你给的验收标准是：

```text
检测尺寸数量：52
查看报表输出尺寸项数

检出率：100%
检测600件良品、不良品，均可输出尺寸结果

检出静态重复性0.1mm的尺寸数量：10
查看报表输出的静态重复性0.1mm的尺寸数量

检出静态重复性0.05mm的尺寸数量：10
查看报表输出的静态重复性0.05mm的尺寸数量

检出静态重复性0.03mm的尺寸数量：32
查看报表输出的静态重复性0.03mm的尺寸数量
```

这里最合理的理解是：

1. 系统一共要输出 52 个尺寸项。
2. 这 52 个尺寸项按重复性要求分成三个等级。
3. 其中 10 个尺寸项要求静态重复性达到 0.1mm。
4. 其中 10 个尺寸项要求静态重复性达到 0.05mm。
5. 其中 32 个尺寸项要求静态重复性达到 0.03mm。

这三个数量正好相加：

```text
10 + 10 + 32 = 52
```

所以它们更像是验收方预先定义好的尺寸项等级，而不是算法自动根据某一次结果随便划出来的等级。

### 1.1 0.1mm、0.05mm、0.03mm不是直接和标准尺寸的差值

如果标准写的是“静态重复性”，通常不是指：

```text
算法测量值 - 标准值 = 0.1mm
```

而是指同一个尺寸在静态条件下重复测量时，测量结果自身的波动不超过某个阈值。例如同一个尺寸连续拍 20 张，得到 20 个长度：

```text
L1, L2, L3, ..., L20
```

一种常见口径是极差：

```text
静态重复性 = max(L1..L20) - min(L1..L20)
```

如果这个值小于等于 0.03mm，就说这个尺寸项的静态重复性满足 0.03mm。

也可能有客户使用标准差、3σ 或 6σ：

```text
标准差口径：std(L1..L20)
3σ口径：3 * std(L1..L20)
6σ口径：6 * std(L1..L20)
```

因此，必须向验收方确认“静态重复性”到底用哪一个公式。当前代码输出了两个有用指标：

- `length_range_px`：极差，也就是最大值减最小值。
- `length_std_px`：标准差。

但当前报表没有把它们换算成毫米，也没有按 0.1/0.05/0.03mm 统计 52 个尺寸项。

### 1.2 当前脚本和验收标准不是一个完整对应关系

验收标准说的是 52 个尺寸项。

当前脚本实际只输出一个长度测量项：

```text
length_px
```

它不是 52 个尺寸项的完整验收报表。当前 JSON 只能用于理解“一条长度测量”的重复性统计方式，不能直接证明 52 个尺寸项全部满足验收。

如果要做完整验收，报表至少应该有：

- 52 个尺寸项的名称或编号。
- 每个尺寸项对应的重复性等级要求：0.1mm、0.05mm 或 0.03mm。
- 每个尺寸项的重复测量数据。
- 每个尺寸项的静态重复性结果，单位为 mm。
- 每个尺寸项是否合格。
- 最后统计三类尺寸项数量是否分别为 10、10、32。

## 2. 当前 JSON 里的重复性是怎么算的

当前代码按路径解析样品和位置：

```text
sample_1/pos_1/... -> sample_id = s1, position = pos1
sample_1/pos_2/... -> sample_id = s1, position = pos2
sample_1/pos_3/... -> sample_id = s1, position = pos3
sample_2/pos_1/... -> sample_id = s2, position = pos1
...
```

然后按 `(sample_id, position)` 分组。

当前数据是：

```text
sample_1: 3 个位置，每个位置 20 张，共 60 张
sample_2: 3 个位置，每个位置 20 张，共 60 张
```

所以 JSON 里有 6 条重复性结果：

```text
s1 + pos1
s1 + pos2
s1 + pos3
s2 + pos1
s2 + pos2
s2 + pos3
```

如果只看一个零件，比如 `sample_1`，就是 3 个位置各做了一次静态重复性统计。所以你理解的“每个零件的三个位置各做一次静态重复性测试”是对的。

当前代码不是把一个零件的 60 张图片混在一起算一个静态重复性。

### 2.1 单张图的长度计算

每张图片先测出两个端点：

```text
(x1, y1)
(x2, y2)
```

然后计算像素长度：

```text
length_px = sqrt((x2 - x1)^2 + (y2 - y1)^2)
```

这个值写入 `measurements.csv` 的 `length_px`。

### 2.2 每个位置 20 张图的重复性计算

对同一个 `(sample_id, position)` 下的 20 个 `length_px`：

```text
length_min_px = min(length_px)
length_max_px = max(length_px)
length_range_px = length_max_px - length_min_px
length_mean_px = mean(length_px)
length_std_px = std(length_px)
```

当前代码里的 `length_range_px` 就是同一组 20 张图片长度的最大值减最小值。

这就是当前 JSON 中“静态重复性”的主要依据。

当前 `output/static_repeatability.csv` 的结果如下：

| sample | position | 张数 | mean px | range px | std px |
|---|---:|---:|---:|---:|---:|
| s1 | pos1 | 20 | 2737.126519 | 0.036946 | 0.013127 |
| s1 | pos2 | 20 | 2733.747456 | 0.077381 | 0.019620 |
| s1 | pos3 | 20 | 2716.188775 | 0.096568 | 0.028399 |
| s2 | pos1 | 20 | 2736.058237 | 0.441362 | 0.094540 |
| s2 | pos2 | 20 | 2734.229291 | 0.084814 | 0.019710 |
| s2 | pos3 | 20 | 2723.582433 | 0.101974 | 0.022301 |

从 `range px` 看，同一个位置内的波动都很小。最大的是：

```text
s2 pos1: length_range_px = 0.441362 px
```

这说明当前算法在“同一个位置连续拍 20 张”的条件下，输出很稳定。

但这只说明静态重复性，不说明换位置后还准，也不说明测量对象一定是真实物理端面。

## 4. px 到 mm 怎么换算

工业界不存在一个固定的：

```text
1 px = ? mm
```

原因是像素当量取决于整套成像系统：

- 相机传感器尺寸。
- 镜头像素分辨率。
- 镜头焦距。
- 工作距离。
- 视野大小 FOV。
- 是否做了畸变校正。
- 物体所在平面是否和标定平面一致。
- 不同位置是否使用同一标定。

正确做法是用已知尺寸标定：

```text
mm_per_px = 已知真实长度(mm) / 图像中对应长度(px)
px_per_mm = 图像中对应长度(px) / 已知真实长度(mm)
```

如果有一个标准件长度为 `L_mm`，在图像里测得为 `L_px`，那么：

```text
mm_per_px = L_mm / L_px
```

当前标注线长度是：

```text
manual_length_px = 2737.127188 px
```

但当前文件里没有这条线对应的真实毫米长度，所以不能从当前 JSON 自动得出 `1px` 等于多少毫米。

如果真实长度已知，例如这条线真实长度是 `20.000mm`，那么：

```text
mm_per_px = 20.000 / 2737.127188 = 0.007307 mm/px
px_per_mm = 2737.127188 / 20.000 = 136.856 px/mm
```

这里的 `20.000mm` 只是示例，不是当前数据里已有的事实。

## 5. 怎么把当前 range px 换算成静态重复性 mm

如果已经知道：

```text
mm_per_px = k
```

那么当前代码的极差口径可以换算为：

```text
静态重复性_mm = length_range_px * k
```

如果验收方使用标准差口径，则换算为：

```text
std_mm = length_std_px * k
```

如果验收方使用 6σ 口径，则换算为：

```text
repeatability_6sigma_mm = 6 * length_std_px * k
```

所以是否满足 0.1mm、0.05mm、0.03mm，取决于两个前提：

1. `k = mm_per_px` 是多少。
2. 静态重复性的验收公式是 `range`、`std`、`3σ` 还是 `6σ`。

如果临时采用之前讨论中提到的近似：

```text
1 px ≈ 0.0072 mm
```

那么当前 6 组的极差换算大约是：

| sample | position | range px | range mm，按 0.0072 mm/px 估算 |
|---|---:|---:|---:|
| s1 | pos1 | 0.036946 | 0.000266 mm |
| s1 | pos2 | 0.077381 | 0.000557 mm |
| s1 | pos3 | 0.096568 | 0.000695 mm |
| s2 | pos1 | 0.441362 | 0.003178 mm |
| s2 | pos2 | 0.084814 | 0.000611 mm |
| s2 | pos3 | 0.101974 | 0.000734 mm |

按这个临时比例看，组内极差都远小于 0.03mm。

但注意：这个结论只针对“每个位置组内 20 张图”的静态重复性；它不能证明跨位置均值差也满足 0.03mm。

## 6. 三个位置的均值差接近 20px，这怎么算

当前每个位置的组内 `range_px` 很小，但是三个位置的 `length_mean_px` 差异很大。

### 6.1 sample_1 的位置间均值差

```text
s1 pos1 mean = 2737.126519 px
s1 pos2 mean = 2733.747456 px
s1 pos3 mean = 2716.188775 px
```

最大均值减最小均值：

```text
2737.126519 - 2716.188775 = 20.937744 px
```

如果按 `1px ≈ 0.0072mm` 临时估算：

```text
20.937744 * 0.0072 = 0.150752 mm
```

### 6.2 sample_2 的位置间均值差

```text
s2 pos1 mean = 2736.058237 px
s2 pos2 mean = 2734.229291 px
s2 pos3 mean = 2723.582433 px
```

最大均值减最小均值：

```text
2736.058237 - 2723.582433 = 12.475804 px
```

如果按 `1px ≈ 0.0072mm` 临时估算：

```text
12.475804 * 0.0072 = 0.089826 mm
```

### 6.3 这不是组内静态重复性

这个 20px 左右的差异不是当前 JSON 里的 `length_range_px`，因为当前 `length_range_px` 只在同一个 `(sample_id, position)` 的 20 张图内部计算。

位置间均值差应该单独叫：

```text
位置间一致性
跨位置再现性
换位置系统偏差
```

它回答的是：

```text
同一个尺寸换到不同位置拍，测出来是否仍然一致？
```

这和静态重复性不是同一个验收指标。

## 7. 这个 20px 差异是否算失败，要看三种场景

### 场景 A：三个 pos 是三个不同的物理测量位置

如果 `pos1`、`pos2`、`pos3` 本来就是零件上三个不同位置，测的不是同一个物理尺寸，那么它们的均值不需要相等。

这时应该分别判断：

```text
pos1 的尺寸是否接近 pos1 的标准值
pos2 的尺寸是否接近 pos2 的标准值
pos3 的尺寸是否接近 pos3 的标准值
```

不能简单要求三个位置的 `length_mean_px` 一样。

### 场景 B：三个 pos 是同一个尺寸的三次换位拍摄

如果 `pos1`、`pos2`、`pos3` 是同一个零件、同一个尺寸，只是放到三个位置或三个视野里重复拍，那么三组均值应该接近。

这时 20px 级别的差异就不是静态重复性问题，而是跨位置再现性或系统误差问题。

这种误差要用下面的公式单独算：

```text
cross_pos_range_px = max(pos_mean_px) - min(pos_mean_px)
cross_pos_range_mm = cross_pos_range_px * mm_per_px
```

然后再和允许的位置间误差比较。

如果按 `1px ≈ 0.0072mm` 临时估算：

```text
s1 跨位置差异 ≈ 0.150752mm
s2 跨位置差异 ≈ 0.089826mm
```

这已经可能超过 0.03mm 或 0.05mm 等高精度要求。是否超过 0.1mm，要看真实 `mm_per_px` 和客户验收口径。

### 场景 C：三个 pos 是同一尺寸，但算法测到的不是同一物理边界

从当前算法现象看，这种风险很高。

当前算法主要检测高亮长条/反光区域的左右边缘，而不是严格检测零件真实端面。不同位置的反光形态变化后，算法可能仍然很稳定地测到同一个反光区域，但这个反光区域本身已经不是同一个物理边界。

这就会出现：

```text
组内 range 很小
跨位置 mean 差很大
```

这类情况说明算法重复性好，但准确性或位置一致性有问题。

换句话说：

```text
算法可以很稳定地测错。
```

## 8. 针对你的三个问题逐条回答

### 问题 1：三个级别是根据什么划分的

这三个级别通常不是算法自己随便划分的，而是验收文件或尺寸清单预先规定的。

合理理解是：

```text
52 个尺寸项中：
10 个尺寸项要求静态重复性 <= 0.1mm
10 个尺寸项要求静态重复性 <= 0.05mm
32 个尺寸项要求静态重复性 <= 0.03mm
```

`0.1mm`、`0.05mm`、`0.03mm` 一般不是“算法测量值和标准值相差多少”，而是“同一静态条件下重复测量的波动不超过多少”。

但最终必须向验收方确认公式：

```text
静态重复性 = max - min ?
静态重复性 = std ?
静态重复性 = 3σ ?
静态重复性 = 6σ ?
```

当前代码更接近 `max - min` 口径，因为 JSON 里明确输出了 `length_range_px`。

### 问题 2：是不是每个零件三个位置各做一次静态重复性测试

按当前 JSON 和代码，是的。

当前统计方式是：

```text
每个 sample
  每个 position
    20 张图片
    计算一次重复性
```

所以：

```text
sample_1 有 3 个 position -> 3 条重复性结果
sample_2 有 3 个 position -> 3 条重复性结果
```

总共就是 6 条。

它不是：

```text
sample_1 的 60 张图片混在一起算一个重复性
sample_2 的 60 张图片混在一起算一个重复性
```

如果把 60 张混在一起，三个位置之间的系统差异也会被混进来，这就不再是严格的“静态重复性”。

### 问题 3：每个 px 对应多少 mm，0.1/0.05/0.03 是针对零件还是位置

`1px` 对应多少 `mm` 没有工业统一值，必须由标定得到。

公式是：

```text
mm_per_px = 标准件真实长度(mm) / 图像测得长度(px)
```

`0.1mm`、`0.05mm`、`0.03mm` 应该针对“尺寸项”的重复性要求，而不是笼统针对一个零件。

如果一个零件有 52 个尺寸项，那么每个尺寸项都有自己的重复性要求。这个尺寸项如果在某个固定位置拍 20 次，就算这个位置下该尺寸项的静态重复性。

如果同一个尺寸项要在三个不同位置都能测，那么应该同时检查两件事：

1. 每个位置内部的静态重复性是否合格。
2. 三个位置之间的均值是否一致，或者是否分别接近各自的标准值。

当前结果看：

- 每个位置组内的 `length_range_px` 都很小，组内静态重复性看起来很好。
- 但不同位置的 `length_mean_px` 差异明显，说明跨位置一致性或算法准确性存在问题。

这两个结论不矛盾。

## 9. 建议验收时明确的口径

为了避免后续争议，验收文档或报表字段最好明确下面这些内容。

### 9.1 尺寸项清单

需要列出 52 个尺寸项，例如：

```text
D01, D02, ..., D52
```

每个尺寸项要标明：

```text
尺寸名称
所在位置
名义值
公差
重复性等级：0.1 / 0.05 / 0.03mm
```

### 9.2 重复性公式

明确使用哪个公式：

```text
range = max - min
std
3σ
6σ
```

如果不明确，算法方和验收方可能都说自己是对的，但数字无法比较。

### 9.3 像素到毫米标定

明确：

```text
mm_per_px
px_per_mm
标定图
标定板或标准件尺寸
是否每个 position 独立标定
是否做畸变校正
```

对于高精度测量，不建议三个位置共用一个简单像素比例，除非已经证明三个位置处于同一成像比例和同一校正平面。

### 9.4 位置间一致性

如果三个 `pos` 是同一个尺寸的不同拍摄位置，必须新增一个指标：

```text
position_mean_range_mm = (max(position_mean_px) - min(position_mean_px)) * mm_per_px
```

这个指标不能混进静态重复性里，但必须单独验收。

### 9.5 准确度

如果要证明算法测得准，还必须和计量标准比：

```text
accuracy_error_mm = algorithm_mean_mm - reference_mm
```

静态重复性再好，也不能替代准确度验收。

## 10. 当前结果的结论

基于当前 JSON，可以得到以下结论：

1. 当前代码按 `sample_id + position` 分组统计重复性。
2. 一个零件三个位置会产生三条重复性结果；两个零件就是六条。
3. 当前 `length_range_px` 是每个位置 20 张图内部的极差。
4. 当前没有真实 `mm_per_px`，所以不能正式判定 0.1/0.05/0.03mm 是否满足。
5. 如果临时按 `1px ≈ 0.0072mm`，组内静态重复性看起来都满足 0.03mm。
6. 但三个位置的 `length_mean_px` 差异明显，`s1` 约 20.94px，`s2` 约 12.48px。
7. 这个差异不是静态重复性，而是跨位置一致性、系统误差或算法测量对象变化。
8. 如果三个位置本来应该测同一个物理尺寸，那么当前跨位置差异需要单独处理，不能只看 `length_range_px`。
9. 如果三个位置本来就是三个不同物理尺寸，则需要分别拿各自的标准值比较，不能要求均值相同。
